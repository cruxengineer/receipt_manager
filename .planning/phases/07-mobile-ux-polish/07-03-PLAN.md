<!--
  Forward flow (slide left — new screen from right):

  Back flow (slide right — screen exits right, previous screen appears):
    summary → swipe (via Adjust)
    review → capture (via Back)

  'processing' is not a screen — CaptureScreen renders for both 'capture' and 'processing'.
  No transition needed between capture ↔ processing (same component, isProcessing prop changes).
-->

<!-- IMPLEMENTATION APPROACH (CSS keyframe + React key remount): -->
<!--
  1. Keep a `transitionClass` state ('slide-forward' | 'slide-back' | 'fade-out' | '')
  2. Keep a `transitionKey` counter that increments on every real screen change
  3. Wrap the rendered screen in a div with key={transitionKey} and className={transitionClass}
  4. On each appState change, determine direction BEFORE calling setAppState,
     then call setTransitionClass + setTransitionKey + setAppState together

  Key insight: React re-renders on key change, which restarts CSS animations automatically.
  The animation plays once (forwards fill: both) and the element stays in final position.
-->

<task type="auto">
  <name>Task 2: Wire transition classes into App.tsx state machine</name>
  <files>src/App.tsx</files>
  <action>
    In App.tsx, add a transition tracking mechanism and wrap all screen renders in an animated div.

    **Step 1: Add transition state** (add after the existing `useState` declarations):
    ```tsx
    const [transitionClass, setTransitionClass] = useState('')
    const [transitionKey, setTransitionKey] = useState(0)
    ```

    **Step 2: Create a helper to advance state with a transition direction**:
    Add this helper function inside the App component (before the handler functions):
    ```tsx
    const navigate = (next: AppState, direction: 'forward' | 'back' | 'fade' | 'none') => {
      const cls =
        direction === 'forward' ? 'screen-slide-forward' :
        direction === 'back'    ? 'screen-slide-back' :
        direction === 'fade'    ? 'screen-fade-out' : ''
      setTransitionClass(cls)
      setTransitionKey(k => k + 1)
      setAppState(next)
    }
    ```

    **Step 3: Update all handler functions to use `navigate()` instead of `setAppState()`**:

    - `handleUnlock`: `navigate('names', 'forward')`
    - `handleNamesConfirm`: `navigate(returnAfterNames, returnAfterNames === 'summary' ? 'back' : 'forward')` — then reset returnAfterNames to 'capture'
    - `handleAdjust`: `navigate('swipe', 'back')` (going back to adjust assignments)
    - `handleStartOver`: call all resets as before, then `navigate('capture', 'fade')`
    - `handleEditNamesFromSummary`: `navigate('names', 'none')` (modal-like, no directional slide)
    - `handleSubmit` on success: `navigate('review', 'forward')`
    - `handleSubmit` on error: `navigate('capture', 'none')` (error state, same visual level)
    - `handleRetry`: `navigate('capture', 'none')`
    - `handleAddAnotherReceipt`: `navigate('capture', 'none')` (adding more, same screen)
    - `handleAddManually`: `navigate('review', 'forward')`
    - `handleBack` (review → capture): `navigate('capture', 'back')`
    - `handleConfirm`: `navigate('swipe', 'forward')`
    - `handleSwipeComplete` (all done → summary): `navigate('summary', 'forward')` — the 1.5s pause already happens in SwipeScreen before onComplete fires

    For the `setAppState('processing')` call inside `handleSubmit` (before the AI call):
    Use `navigate('processing', 'none')` — processing is not a screen change visually.

    **Step 4: Wrap screen renders in animated div**:

    Replace the current conditional render block (the if/return chain) with a single animated wrapper:

    ```tsx
    const screenContent = (() => {
      if (appState === 'gate') {
        return <PasswordGate onUnlock={handleUnlock} />
      }
      if (appState === 'names') {
        return (
          <NamesModal
            defaultNameA={personAName}
            defaultNameB={personBName}
            onConfirm={handleNamesConfirm}
          />
        )
      }
      if (appState === 'review') {
        return (
          <ReviewScreen
            items={reviewItems}
            skippedRegions={skippedRegions}
            sourceFiles={sourceFiles}
            aiAttempted={aiAttempted}
            receiptGroupSizes={receiptGroupSizes}
            onConfirm={handleConfirm}
            onBack={handleBack}
            onAddAnother={handleAddAnotherReceipt}
          />
        )
      }
      if (appState === 'swipe') {
        return (
          <SwipeScreen
            key={swipeKey}
            items={confirmedItems}
            personAName={personAName}
            personBName={personBName}
            onComplete={handleSwipeComplete}
          />
        )
      }
      if (appState === 'summary') {
        return (
          <SummaryScreen
            assignments={assignments}
            personAName={personAName}
            personBName={personBName}
            onAdjust={handleAdjust}
            onStartOver={handleStartOver}
            onEditNames={handleEditNamesFromSummary}
          />
        )
      }
      // appState === 'capture' | 'processing'
      return (
        <CaptureScreen
          isProcessing={appState === 'processing'}
          error={error}
          onRetry={handleRetry}
          onAddManually={handleAddManually}
          onSubmit={handleSubmit}
        />
      )
    })()

    return (
      <div
        key={transitionKey}
        className={transitionClass}
        style={{ willChange: transitionClass ? 'transform, opacity' : undefined }}
      >
        {screenContent}
      </div>
    )
    ```

    Important notes:
    - The `key={transitionKey}` on the outer div causes React to remount it on each navigation, restarting the CSS animation fresh.
    - Keep the existing `useState` lazy initializer for appState (`sessionStorage.getItem(SESSION_KEY)` check) — do not change it.
    - Keep the existing `swipeKey` logic in handleAdjust/handleStartOver — SwipeScreen still needs key remounting for its own internal state reset.
    - The `willChange` hint tells the GPU to composite this layer, making animation smoother on mobile.
    - Do NOT wrap in `overflow-hidden` at the App level — screens use `h-dvh overflow-hidden` internally. Wrapping at App level would clip during slide transitions.
